# Phase 21.1: Init-led MCP onboarding and read-only status contract - Context

**Gathered:** 2026-03-20
**Status:** Ready for planning

<domain>
## Phase Boundary

Correct the Phase 21 command-ownership mistake by making `optimusctx init` the canonical supported-client preview and write surface, while restoring `optimusctx status` to a read-only runtime report.

</domain>

<decisions>
## Implementation Decisions

### Command Ownership
- `optimusctx init --client <client> [--write]` owns supported-client MCP preview and write onboarding.
- Claude CLI scope selection moves onto `init` so the onboarding surface still covers `local`, `project`, and `user`.
- `optimusctx status` stops owning MCP registration preview or write behavior and returns to pure runtime/status reporting.

### Compatibility Shape
- Keep the shared `InstallService` registration backend and reuse it from `init` rather than introducing a second registration path.
- Keep `optimusctx install` as a deprecated compatibility shim, but retarget its warning/help toward the init-led onboarding contract.
- Preserve `optimusctx run` as the canonical MCP runtime handoff across every supported host.

### Operator Guidance
- Supported-client guidance must consistently name `claude-desktop`, `claude-cli`, `codex-app`, and `codex-cli`.
- In-product guidance that previously pointed at `status --client ...` should point at `init --client ...` instead.
- Public documentation rewrites stay deferred to Phase 22; this phase only fixes command/help/operator-surface truthfulness in the product and tests.

### Claude's Discretion
Implementation details inside the existing CLI/app seams are at Claude's discretion as long as the command-ownership contract above is preserved.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `internal/app/install.go` already provides the shared supported-client preview/write backend used by `init`, `status`, and `install`.
- `repository.NormalizeClaudeCLIScope` and the Codex config helpers already cover the host-specific write semantics Phase 21.1 needs to preserve.
- Existing CLI tests in `internal/cli/init_*`, `internal/cli/status_test.go`, and `internal/cli/doctor_test.go` provide direct seams for locking the new contract.

### Established Patterns
- CLI commands parse flags locally, then forward typed `app.InstallRequest` values into the app layer.
- Operator copy is regression-locked with exact-string tests rather than indirect snapshot fixtures.
- Deprecated/operator surfaces keep `optimusctx run` visible even when the registration entrypoint changes.

### Integration Points
- `internal/cli/init.go` becomes the main supported-client onboarding surface.
- `internal/cli/status.go` must shed registration flags and keep only read-only runtime health output.
- `internal/cli/install.go`, `internal/app/snippet.go`, and `internal/app/doctor.go` must align with the init-led contract so Phase 22 docs are built on truthful command behavior.

</code_context>
